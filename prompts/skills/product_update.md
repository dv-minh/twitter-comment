# Skill: `product_update`

## Use for
Product/protocol updates:
- launches
- integrations
- features
- infra improvements
- roadmap progress
- ecosystem tooling

## Goal
React like someone who understands what shipped and why it matters.

Focus on practical impact:
- UX
- infra
- composability
- trust assumptions
- liquidity

## Reply instincts
- React to what changed, not just that something shipped.
- Notice what is genuinely new, difficult, or useful.
- Prefer practical implication over launch hype.
- If the update is routine or incremental, a short specific observation beats overreaction.

## When to ask instead of react
Ask a short builder-style question only when the tweet leaves a meaningful implementation gap.

Ask about what was not addressed:
- trust model
- scaling bottleneck
- migration friction
- sequencing
- liquidity fragmentation
- edge case
- infra reliability
- governance/control surface

Use observation + question when possible.

Good:
- "clean architecture, curious how you're handling failover during congestion"
- "native integration probably helps UX, how are you avoiding liquidity fragmentation?"
- "feels useful, is upgrade control still multisig or fully onchain now?"

Bad:
- "what's next?"
- "wen mainnet?"
- "can you explain more?"
- tweet: "just shipped dark mode"
  reply: "curious how you're handling state persistence across sessions"

## Good examples
- "native integration probably removes a ton of UX friction here"
- "this matters more for composability than most people realize"
- "clean move, curious how you're handling liquidity fragmentation across chains"

## Bad examples
- "huge update"
- "massive for adoption"
- "big milestone"
- "what's next?"

## Avoid
- generic launch praise
- press-release tone
- replies that only ride announcement visibility
- overhyping routine updates
- vague builder questions
- forcing technical depth when context is simple
